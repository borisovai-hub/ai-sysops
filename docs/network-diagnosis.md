# Где именно проблема в сети и как исправить

Пошаговая диагностика с ПК пользователя (где сайт не открывается или грузится 20+ секунд). Цель — найти участок: **DNS → маршрут → MTU → сервер** и применить нужное исправление.

**Скрипт диагностики (Windows, PowerShell):** из корня репозитория запустите `scripts\diagnose-network.ps1`. Можно указать домен и URL:  
`.\scripts\diagnose-network.ps1 -Domain "borisovai.tech" -Url "https://borisovai.tech/ru/blog/..."`

---

## Схема: где может быть сбой

```
[Ваш ПК] → DNS → [Провайдер] → [Интернет] → [Сервер]
     ↑         ↑         ↑              ↑
   шаг 0     шаг 1      шаг 2        шаг 3
```

| Участок | Что проверяем | Где править |
|---------|----------------|-------------|
| **0. ПК** | Кэш DNS, прокси, MTU интерфейса | ПК (hosts, DNS, настройки) |
| **1. DNS** | Время и результат разрешения имени | ПК (смена DNS), реже — NS у регистратора |
| **2. Маршрут / MTU** | Задержка, потери, размер пакетов по пути | Сервер (MSS, GZIP), реже — только обход |
| **3. Сервер** | Доступность порта, ответ HTTP | Сервер (Traefik, бэкенд) |

---

## Шаг 1. Проверка DNS (участок 1)

**На проблемном ПК** откройте PowerShell или cmd.

```powershell
# Время разрешения имени (подставьте свой домен)
Measure-Command { Resolve-DnsName borisovai.tech -Server 8.8.8.8 } | Select-Object TotalSeconds
```

- **< 1 с** — DNS тут вряд ли причина.
- **2–10 с и больше** — DNS в этой сети медленный или нестабильный.

**Исправление:** сменить DNS на этом ПК на 8.8.8.8 или 77.88.8.8 (Параметры → Сеть → адаптер → IPv4 → DNS). Затем:

```powershell
Clear-DnsClientCache
```

Проверить, что возвращается нужный IP:

```powershell
Resolve-DnsName borisovai.tech -Server 8.8.8.8 | Format-Table Name, Type, IPAddress -AutoSize
```

Если IP неверный или ошибка — в этой сети блокируют/подменяют DNS → обход через **hosts** (см. [DNS_TROUBLESHOOTING.md](../DNS_TROUBLESHOOTING.md)).

## Шаг 2. Проверка задержки и потерь (участок 2)

```powershell
ping -n 10 borisovai.tech
```

Смотрите:

- **Потеря пакетов (loss)** — нестабильный канал или блокировка ICMP по пути.
- **RTT (время ответа)** — если стабильно > 150–200 ms, страница будет грузиться с задержкой даже при нормальном MTU.

**Исправление:** высокий RTT или потери на уровне провайдера/маршрута — поменять можно только сеть (другой Wi‑Fi, мобильный интернет, VPN) или перенести сервер/CDN ближе к пользователю. На сервере уже должны быть включены GZIP и MSS (см. ниже).

---

## Шаг 3. Проверка MTU по пути (участок 2)

Если **ping без размера** проходит, а **сайт в браузере** грузится 15–20 с или обрывается — часто виноват Path MTU (где‑то на пути режут большие пакеты).

**Проверка с ПК (Windows):**

```powershell
# Пакет ~1500 байт (как типичный TCP сегмент) — может не пройти в «узкой» сети
ping -n 2 -l 1472 borisovai.tech

# Пакет маленький — обычно проходит
ping -n 2 -l 500 borisovai.tech
```

- **1472 не проходит (таймаут), 500 проходит** — на пути уменьшенный MTU; большие пакеты режут или дропают.
- **Оба проходят** — по ICMP путь в порядке, но для TCP проблема может остаться (не все маршрутизаторы обрабатывают ICMP «Fragmentation needed»).

**Исправление на сервере (обязательно):** уменьшить размер исходящих TCP-пакетов и включить сжатие:

```bash
sudo ./scripts/single-machine/fix-mtu-issue.sh
```

Скрипт выставляет `tcp_mtu_probing` и **TCP MSS 1360** для всех исходящих соединений. Плюс GZIP в Traefik (site-compress) — ответы меньше, меньше пакетов. После этого повторить загрузку страницы с проблемного ПК.

## Шаг 4. Проверка HTTP: кто тормозит — сеть или сервер

**На проблемном ПК** (PowerShell, подставьте свой URL):

```powershell
# С GZIP (как браузер) — ожидаем < 2–3 с
curl.exe -s -o NUL -w "GZIP: %{time_total}s size %{size_download}\n" -H "Accept-Encoding: gzip" "https://borisovai.tech/ru/blog/feat-add-global-ckeditor-compatible-content-styles-across-all-sections-9weupr"

# Без GZIP — в «узкой» сети часто 15–20 с
curl.exe -s -o NUL -w "No GZIP: %{time_total}s\n" "https://borisovai.tech/ru/blog/feat-add-global-ckeditor-compatible-content-styles-across-all-sections-9weupr"
```

Интерпретация:

| GZIP быстрый | Без GZIP | Вывод | Что делать |
|--------------|----------|--------|------------|
| Да (< 3 с)   | Медленно (15–20 с) | Проблема в размере пакетов на пути (MTU) | На сервере: `fix-mtu-issue.sh`; у пользователя — убедиться, что браузер отправляет `Accept-Encoding: gzip` (без прокси/расширений, которые его снимают). |
| Оба медленные | Оба медленные | Высокий RTT или потери на маршруте | Смена сети, VPN или хостинг ближе к пользователю. |
| Оба быстрые   | — | С этой машины сеть в порядке | Искать причину в браузере (расширения, кэш) или в другом устройстве/сети. |

## Шаг 5. Проверка сервера (участок 3)

Если с **другого** ПК/сети сайт открывается нормально, а с проблемного — нет, причина в сети или на проблемном ПК (шаги 1–4). Если **нигде** не открывается — смотреть сервер.

**На сервере:**

```bash
# Traefik слушает 443
sudo ss -tlnp | grep 443

# Ответ бэкенда сайта (порт из install-config: site_port, например 4001)
curl -sI -H "Accept-Encoding: gzip" http://127.0.0.1:4001/ru/blog/feat-add-global-ckeditor-compatible-content-styles-across-all-sections-9weupr | head -5
```

Если локально ответ 200 и быстрый — проблема не в приложении, а в сети до сервера (шаги 1–4).

## Сводная таблица: где проблема и как исправить

| Симптом | Где искать | Как исправить |
|--------|------------|----------------|
| Долго резолвится имя или неверный IP | DNS (участок 1) | Смена DNS на 8.8.8.8 / 77.88.8.8, очистка кэша; при блокировке — hosts. |
| Пинг с потерей или очень большой RTT | Маршрут (участок 2) | Другая сеть; на сервере — GZIP + MSS уже должны быть. |
| ping -l 1472 не проходит, -l 500 проходит | MTU на пути (участок 2) | На сервере: `sudo ./fix-mtu-issue.sh` (MSS 1360 + GZIP). |
| curl с GZIP быстрый, без GZIP — 20 с | MTU + размер ответа | Сервер: fix-mtu-issue.sh; клиент: не снимать Accept-Encoding (прокси/антивирус). |
| В браузере 20 с, curl с GZIP быстрый | Браузер не шлёт GZIP или тормозят шрифты/JS | [site-not-accessible-some-networks.md](site-not-accessible-some-networks.md) раздел 4.4 «Тест в порядке, браузер тормозит»: DevTools → Сеть → кто долгий; проверить Accept-Encoding; отключить прокси/расширения или другой браузер. |
| Нигде не открывается | Сервер или DNS глобально | Проверить Traefik, бэкенд (шаг 5), DNS у регистратора. |

## Быстрый чек-лист на проблемном ПК

1. `Clear-DnsClientCache`; проверить `Resolve-DnsName borisovai.tech -Server 8.8.8.8`.
2. `ping borisovai.tech` — смотреть потери и RTT.
3. `ping -l 1472 borisovai.tech` vs `ping -l 500 borisovai.tech`.
4. `curl.exe -s -o NUL -w "%{time_total}s\n" -H "Accept-Encoding: gzip" "https://borisovai.tech/ru/blog/..."`.
5. На сервере: `sudo ./scripts/single-machine/fix-mtu-issue.sh` (если ещё не запускали).

См. также: [site-not-accessible-some-networks.md](site-not-accessible-some-networks.md), [DNS_TROUBLESHOOTING.md](../DNS_TROUBLESHOOTING.md).
