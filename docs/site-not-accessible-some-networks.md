# Сайт недоступен из части сетей

Если https://borisovai.tech (или другой домен за Traefik) открывается в одной сети, но не открывается в другой (таймаут, «нет соединения», обрыв при загрузке страницы), причина обычно одна из двух: **Path MTU** или **DNS**.

**Найти, где именно проблема в сети (DNS → маршрут → MTU → сервер), и что исправить:** пошаговая инструкция в [network-diagnosis.md](network-diagnosis.md).

---

## 1. Path MTU Discovery (чёрная дыра MTU)

В части сетей (мобильный интернет, корпоративные VPN, провайдеры с уменьшенным MTU) пакеты большого размера отбрасываются, соединение «висит» или обрывается. Особенно заметно на длинных страницах и больших ресурсах (JS/CSS).

### Что сделать на сервере

Выполните один раз (из директории со скриптами на сервере):

```bash
sudo ./scripts/single-machine/fix-mtu-issue.sh
```

Скрипт:
- Включает `net.ipv4.tcp_mtu_probing = 1` (применяется ко всем исходящим TCP).
- **Принудительно ограничивает TCP MSS (1360)** для всех исходящих соединений через iptables — пакеты меньше, стабильнее в «узких» сетях.
- Для Mailu добавляет compress в `mailu.yml`; для сайта GZIP уже задан в `site.yml`. Отключить только MSS: `sudo ./fix-mtu-issue.sh --no-iptables`.

Подробнее: [server-fix-instructions.md](server-fix-instructions.md).

## 2. DNS в «проблемной» сети

В другой сети может использоваться другой DNS (провайдер, корпоративный, блокирующий). Домен может не резолвиться или отдавать старый/неверный IP.

### На клиенте (ПК в сети, где не открывается)

1. **Очистить кэш DNS** (PowerShell от администратора):
   ```powershell
   Clear-DnsClientCache
   ```

2. **Проверить разрешение имени**:
   ```powershell
   Resolve-DnsName borisovai.tech -Server 8.8.8.8
   Resolve-DnsName borisovai.tech -Server 77.88.8.8
   ```
   Убедитесь, что возвращается нужный A-адрес сервера (тот же, что в рабочей сети).

3. **Временно подставить IP в hosts** (если DNS в сети «ломает» домен):
   - Файл: `C:\Windows\System32\drivers\etc\hosts` (открыть от имени администратора).
   - Добавить строку (IP замените на реальный IP вашего сервера):
     ```
     144.91.108.139    borisovai.tech www.borisovai.tech
     ```
   - Сохранить и снова открыть https://borisovai.tech/ru/blog/...

4. **Сменить DNS на клиенте** на стабильный: 77.88.8.8, 77.88.8.1 (Яндекс) или 8.8.8.8, 8.8.4.4 (Google). Параметры → Сеть → свойства адаптера → IPv4 → указать предпочтительный/альтернативный DNS.

Полный чек-лист DNS: [../DNS_TROUBLESHOOTING.md](../DNS_TROUBLESHOOTING.md).

| Где | Действие |

## 4. Если страница открывается, но очень долго (20+ секунд)

Проверьте по шагам, **что именно** тормозит.

### 4.1. Какой запрос тормозит

В браузере откройте DevTools → вкладка **Сеть** (Network), обновите страницу. Отсортируйте по столбцу «Время» или посмотрите полосу загрузки (waterfall):

- Если **один запрос** висит 15–20 с — это кандидат (часто это HTML без GZIP или крупный ресурс: шрифт, JS).
- Если **много запросов** по 2–5 с — проблема в пути (MTU, RTT) или в количестве ресурсов.

### 4.2. Возможные причины

| Причина | Как проверить | Что сделать |
|--------|----------------|-------------|
| **Клиент не отправляет GZIP** | В Network по клику по первому запросу (документ) → «Заголовки» → «Запрос»: есть ли `Accept-Encoding: gzip`. | Отключить прокси/VPN/антивирус, который убирает заголовок; попробовать другой браузер. |
| **Шрифты (.woff2) или крупный JS** | В Network найти запросы с большим временем к `_next/static/` (шрифты, чанки). | MTU/MSS на сервере уже уменьшают пакеты; в «узкой» сети задержка может остаться. Можно отложить загрузку шрифтов (на стороне сайта). |
| **Медленный DNS у пользователя** | В командной строке: `nslookup borisovai.tech` — замерить время. | Сменить DNS на 77.88.8.8 или 8.8.8.8 (см. п. 2). |
| **Большая задержка (RTT) до сервера** | `ping borisovai.tech` — смотреть время ответа. | Это сетевая география; ускорить можно только CDN или сервер ближе к пользователю. |
| **Прокси/файрвол режет или сканирует трафик** | Та же сеть без прокси (другой браузер, мобильный интернет) — быстрее? | Использовать прямое подключение или другую сеть. |

### 4.3. Проверка с вашего ПК (PowerShell)

Убедиться, что сервер отдаёт страницу с GZIP быстро:

```powershell
# С GZIP (как браузер) — должно быть быстро (< 2 с)
curl.exe -s -o NUL -w "Time: %{time_total}s Size: %{size_download}\n" -H "Accept-Encoding: gzip" "https://borisovai.tech/ru/blog/..."

# Без GZIP — может быть 15–20 с из-за MTU
curl.exe -s -o NUL -w "Time: %{time_total}s\n" "https://borisovai.tech/ru/blog/..."
```

Если с GZIP быстро, а в браузере у пользователя медленно — у пользователя либо не отправляется `Accept-Encoding: gzip`, либо тормозят другие ресурсы (шрифты, скрипты).

### 4.4. Тест в порядке, браузер тормозит — что проверить

Когда curl/скрипт показывают быструю загрузку с GZIP, а в браузере страница грузится 15–20+ секунд, причина почти всегда одна из двух: **браузер не отправляет GZIP** или **тормозят не документ, а другие запросы** (шрифты, JS).

**Шаг 1. Узнать, кто тормозит**

1. Откройте сайт в браузере.
2. **F12** → вкладка **Сеть** (Network).
3. Включите **«Сохранить журнал»** (Preserve log), обновите страницу (**F5**).
4. В таблице запросов нажмите на заголовок столбца **«Время»** (Time), чтобы отсортировать по убыванию.
5. Посмотрите **самый долгий запрос** (имя и URL).

- Если самый долгий — **первый запрос** (документ, имя страницы или `borisovai.tech`) на 15–20 с → переходите к шагу 2.
- Если самые долгие — запросы к **`_next/static/`** (шрифты `.woff2`, файлы `.js`) по 5–10 с каждый → переходите к шагу 3.

**Шаг 2. Браузер не отправляет GZIP**

1. В **Сети** кликните по **первому запросу** (документ).
2. Справа откройте **«Заголовки»** (Headers) → блок **«Заголовки запроса»** (Request Headers).
3. Найдите **`Accept-Encoding`**.

- **Нет `Accept-Encoding: gzip`** (или заголовка вообще нет) — какой-то прокси, антивирус или расширение убирает его. Тогда сервер отдаёт ответ без сжатия → большие пакеты → 20 с.
- **Что сделать:** отключить по очереди прокси (Настройки → Сеть → прокси), VPN, антивирус (временное отключение проверки HTTPS), расширения браузера. Либо открыть сайт в **режиме инкогнито** (без расширений) или в **другом браузере** (Chrome / Edge / Firefox) и сравнить. Если в инкогнито/другом браузере быстро — виновато расширение или настройка основного браузера.

**Шаг 3. Тормозят шрифты или JS**

Если долгие запросы — к `_next/static/` (шрифты, чанки JS), это та же проблема MTU на пути: крупные бинарные файлы грузятся медленно. На сервере уже должны быть MSS и GZIP (они помогают документу и текстовым ресурсам); для уже сжатых `.woff2` выигрыш меньше.

- **Со стороны сервера:** больше ничего типового нет; можно лишь убедиться, что `fix-mtu-issue.sh` выполнен.
- **Со стороны сайта (исходный код):** можно отложить загрузку шрифтов (`font-display: optional` или не подгружать все шрифты сразу), чтобы страница становилась интерактивной быстрее, даже если шрифты ещё грузятся.

**Кратко:** тест в порядке, браузер тупит → в DevTools Сеть смотрим самый долгий запрос; если это документ — проверяем Accept-Encoding и снимаем прокси/расширения; если шрифты/JS — это MTU по пути, на сервере уже fix-mtu-issue.sh, при необходимости правим загрузку шрифтов в коде сайта.

### 4.5. Ошибка ERR_HTTP2_PING_FAILED (200 OK, но «network error»)

В консоли браузера видно `GET ... net::ERR_HTTP2_PING_FAILED 200 (OK)` и «Uncaught TypeError: network error». Запрос формально 200, но соединение по HTTP/2 браузер считает мёртвым: проверка живучести (PING) не прошла вовремя.

**Причины:** задержка или обрыв на пути (браузер шлёт HTTP/2 PING, ответ не успевает или теряется); прокси/файрвол режет или задерживает HTTP/2-кадры; долгая загрузка в «узкой» сети (MTU, потери).

**Что сделать:**

1. **На клиенте:** попробовать режим инкогнито, другой браузер (Firefox ведёт себя иначе по HTTP/2); отключить VPN/прокси/антивирус.
2. **На сервере (если ошибка повторяется у многих):** отключить HTTP/2 и оставить только HTTP/1.1 — тогда PING по HTTP/2 не используется, ошибка уходит.
   - Выполнить скрипт (из директории со скриптами):  
     `sudo ./scripts/single-machine/disable-http2-traefik.sh`  
     Скрипт создаёт `/etc/traefik/dynamic/tls-options.yml` с `alpnProtocols: [http/1.1, acme-tls/1]` и перезапускает Traefik.
   - Либо вручную: создать файл `/etc/traefik/dynamic/tls-options.yml` с содержимым:
     ```yaml
     tls:
       options:
         default:
           alpnProtocols:
             - http/1.1
             - acme-tls/1
     ```
     затем `sudo systemctl restart traefik`.  
   После этого клиенты подключаются по HTTP/1.1; мультиплексирование HTTP/2 отключится, но соединения часто стабильнее на «плохих» сетях. Чтобы снова включить HTTP/2 — удалить `tls-options.yml` и перезапустить traefik.
