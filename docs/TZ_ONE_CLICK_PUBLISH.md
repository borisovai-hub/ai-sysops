# Техническое задание: публикация проектов в один клик и масштабирование инфраструктуры

## 1. Цели и границы

**Цель:** после загрузки кода в GitLab новый проект автоматически появляется на сайте: настроены маршруты Traefik, DNS, CI/CD и при успешном пайплайне — деплой без ручных шагов.

**Ограничения:** не менять базовый стек (Traefik, GitLab, текущий DNS API / `manage-dns`), максимально опираться на существующие скрипты и Management UI.

**Результат ТЗ:** один согласованный документ (этот файл), на основе которого можно разбить работу на этапы и задачи.

## 2. Текущее состояние и целевая структура по серверам

### 2.1 Текущее состояние

- **Один сервер:** Traefik (file provider, `/etc/traefik/dynamic`, watch), GitLab, n8n, Mailu, Management UI, DNS API (Cloudflare / DigitalOcean / local), CLI `manage-dns`.
- **Деплой одного проекта (borisovai-site):** [setup-cicd.sh](../scripts/single-machine/setup-cicd.sh) создаёт `/var/www/borisovai-site/{frontend,backend}`, один Runner; [configure-traefik-deploy.sh](../scripts/single-machine/configure-traefik-deploy.sh) — два файла в `dynamic/` и при необходимости DNS.
- **Новый «произвольный» сервис:** вручную [deploy-service.sh](../scripts/vm1-traefik/deploy-service.sh) или через Management UI [POST /api/services](../management-ui/server.js) (DNS + YAML в `dynamic/` + `systemctl reload traefik`).
- **Проблема:** для каждого нового проекта вручную нужны: свой `.gitlab-ci.yml`, переменные CI (в т.ч. `DEPLOY_PATH`), конфиг Traefik, DNS, директории на сервере. Нет единого входа «создал проект в GitLab — он сразу участвует в деплое».

### 2.2 Целевая модель: один проект инфраструктуры для всех серверов

- **Один репозиторий описывает конфигурацию всех серверов.** Отдельного GitLab-проекта на каждую VM нет: правки и исправления вносятся в одном месте, перенос изменений между «проектами» не требуется — всё в одном репо.
- **Общая инфраструктура:** в этом же репозитории хранятся общие скрипты (Traefik, DNS, установка Runner), шаблоны CI для приложений (include), единые соглашения по именованию доменов и путям деплоя.
- **Типы репозиториев:**
  - **Проекты-приложения** (сайты, API, фронты) — отдельные репо с кодом приложения; их CI подключает include из единого проекта инфраструктуры и деплоит на выбранный хост (тег Runner или переменная).
  - **Единый проект инфраструктуры** — один репозиторий (например `borisovai-admin` или выделенный `infra`): общие скрипты, шаблоны CI, конфиг по каждому серверу в подкаталогах (см. п. 2.3). Добавление новой VM = новый подкаталог в этом репо + установка Runner на VM с уникальным тегом.
- Масштабирование: новый сервер добавляется каталогом и конфигом в едином проекте; Runner на новой VM регистрируется с тегом; приложения выбирают хост по тегу Runner из списка в оркестраторе.

### 2.3 Структура единого проекта инфраструктуры

Чтобы один проект под все серверы оставался управляемым, в ТЗ зафиксирована следующая структура (репозиторий — `borisovai-admin` или отдельный `infra`):

```
<repo>/
├── .gitlab-ci.yml              # при необходимости: пайплайн для применения конфигов по серверам
├── config/
│   ├── single-machine/         # уже есть: общий конфиг для single-machine
│   └── servers/                 # конфиг по серверам (один каталог на хост)
│       ├── vm1/                 # первый хост (например текущая машина)
│       │   ├── env.yml          # переменные: internal_ip, runner_tag, base_domain
│       │   └── traefik/         # опционально: фрагменты dynamic только для этого хоста
│       ├── vm2/
│       │   ├── env.yml
│       │   └── ...
│       └── README.md            # описание серверов и как добавлять новый
├── scripts/                    # общие скрипты (уже есть: install-*, setup-*, deploy-service, manage-dns)
│   ├── single-machine/
│   └── ...
├── templates/                  # шаблоны CI для приложений (include)
│   ├── app-deploy.gitlab-ci.yml
│   ├── docs-publish.gitlab-ci.yml
│   ├── product-page.gitlab-ci.yml
│   └── ...
└── docs/                       # документация, в т.ч. это ТЗ
```

- **Общее:** всё, что не привязано к конкретному хосту (скрипты, шаблоны CI, общая документация), лежит в корне или в `scripts/`, `templates/`, `docs/`. Исправление ошибки в скрипте или шаблоне делается один раз — все серверы и приложения используют обновлённый код.
- **По серверу:** в `config/servers/<id>/` хранится только то, что отличается от хоста к хосту: `env.yml` (IP, тег Runner, домен), при необходимости — локальные фрагменты Traefik. Добавление VM = создание каталога `config/servers/vmN/` и заполнение `env.yml`; оркестратор и CI могут читать список серверов по наличию подкаталогов.
- **Секреты:** не хранить в репозитории; только переменные GitLab CI или внешнее хранилище, ссылки в `env.yml` только на имена переменных.

При необходимости единый пайплайн в этом репо может по изменённым путям (например `config/servers/vm2/**`) применять конфиг только к нужному хосту (через SSH или API). Для MVP достаточно ручного применения или запуска скриптов из нужного каталога.

## 3. Целевой сценарий «один клик»

Идеальный поток **для проектов-приложений**:

1. Пользователь создаёт проект в GitLab (или использует шаблон) и пушит код приложения.
2. При первом пуше (или по явному действию «Зарегистрировать проект для деплоя») запускается процесс онбординга.
3. Онбординг: выделение имени/домена проекта, создание DNS записей (`manage-dns`), создание директорий деплоя, генерация/подстановка конфига Traefik, регистрация/привязка CI (шаблон из единого проекта инфраструктуры, переменные, при мульти-сервере — выбор целевого хоста по тегу Runner).
4. При последующих пушах работает обычный CI/CD: build → deploy на целевой хост (тот же сервер или другая VM по тегу Runner).
5. Проект доступен по своему домену (например `project-name.<base-domain>`) с автоматическим SSL.

**Для новых серверов (VM):** добавление хоста не создаёт новый GitLab-проект. В **едином проекте инфраструктуры** создаётся каталог `config/servers/<vm-id>/` с `env.yml` (IP, тег Runner, домен); на новой VM устанавливается Runner с этим тегом. Приложения при регистрации выбирают целевой хост из списка (теги Runner или идентификаторы из `config/servers/`). Исправления в скриптах и шаблонах вносятся один раз в единый репо — перенос между проектами не нужен.

Варианты «одного клика» для MVP:

- **Вариант A:** кнопка в Management UI «Зарегистрировать проект для деплоя»: ввод URL репозитория GitLab (или выбор из списка через GitLab API), домен/поддомен → один запрос выполняет DNS + Traefik + при необходимости создаёт в репозитории default branch файл `.gitlab-ci.yml` из шаблона и CI variables через GitLab API.
- **Вариант B:** webhook от GitLab (например, при создании проекта или при первом push) на наш «Deploy Orchestrator»; оркестратор по правилам (например, по группе/тегу) создаёт DNS, Traefik, директории и подставляет в проект шаблон CI.

**Зафиксировано в ТЗ:** для MVP выбран **Вариант A** (кнопка в UI); на следующем шаге — гибрид с webhook для выбранной группы/тега (раздел 3C).

## 3A. Детальные целевые сценарии по типам проектов

В зависимости от типа продукта результат публикации разный: только исходники и документация, работающее приложение с DNS, инфраструктура «под деплой позже» или продуктовая страница с артефактами для установки. Ниже — сценарии, что автоматизировать в каждом случае, плюсы и минусы.

### Сценарий 1: Исходный код и документация (publish only)

**Цель:** проект публикуется в виде репозитория + документации (сайт документации или страница с ссылкой на репозиторий). Никакого запущенного сервиса, никакого отдельного домена под приложение.

**Что делается автоматически по инструкции/шаблону:**

- Репозиторий в GitLab уже есть или создаётся; доступ (public/internal) настраивается.
- При необходимости — сборка и публикация **документации** (например, GitLab Pages на `project-name.<base>.pages` или статический сайт под тем же основным доменом).
- DNS: только если нужен отдельный поддомен для docs (например `docs.project.<base>`); иначе — без новых DNS-записей.
- Traefik: не нужен для «только репо»; для Pages/статики — один маршрут на статику, если хостим сами.
- CI: шаблон «build docs + publish» (например, Docusaurus/VuePress → `public/` → Pages или в заданную директорию).

**Плюсы:** минимум инфраструктуры, низкий риск, подходит для библиотек, утилит без установщика, внутренних репозиториев.  
**Минусы:** пользователь не получает «живое» приложение или страницу продукта с загрузками — только код и доки.

**Цепочка публикации документации на сайте (без деплоя приложения):** чтобы документы из репозитория действительно **появлялись на основном сайте**, нужна явная цепочка от пуша до отображения. Её необходимо заложить в ТЗ и реализацию:

1. **Источник:** репозиторий проекта с документацией (Markdown, MkDocs, Docusaurus, VuePress, VitePress и т.п.) в заданной структуре (например `docs/` или корень с конфигом генератора).
2. **Триггер:** push в выбранную ветку (например `main`/`docs`) запускает пайплайн по шаблону «только документация».
3. **Сборка в CI:** job собирает документацию (например `mkdocs build` / `npm run docs:build`) в артефакт (каталог `site/` или `public/`).
4. **Публикация на сайт:** собранная статика должна попасть туда, откуда **основной сайт** её отдаёт пользователю. Варианты:
   - **Вариант A (рекомендуемый, зафиксирован в ТЗ):** CI деплоит вывод в фиксированный каталог на сервере основного сайта, например `/var/www/<main-site>/public/docs/<project-slug>/`. Основной сайт настроен отдавать путь `/docs/<project-slug>/` из этого каталога (статикой или через свой роутинг). Никакого отдельного домена и Traefik для доков не нужно — всё в рамках основного сайта.
   - **Вариант B:** отдельный поддомен (например `docs.<base>` или `docs-<project>.<base>`) с одной записью в Traefik на статический каталог или сервис; CI деплоит в каталог, который обслуживает этот маршрут. Документы «на сайте» в смысле того же базового домена.
   - **Вариант C:** GitLab Pages под своим URL; основной сайт содержит каталог проектов и ссылки «Документация» → URL Pages. Документы «появляются» на сайте как ссылки; сам контент живёт на Pages.
5. **Отображение на основном сайте:** в любом варианте основной сайт должен предоставлять пользователю точку входа к документации проекта:
   - либо маршрут вида `/docs/<project-slug>/` или `/projects/<slug>/docs` (Варианты A/B),
   - либо страница/карточка проекта со ссылкой «Документация» (Вариант C или дополнение к A/B).
6. **Регистрация в оркестраторе:** при выборе сценария «только исходники и документация» оркестратор создаёт/привязывает CI (шаблон «build docs + publish»), задаёт `project-slug` и целевой путь публикации (или URL Pages); при вариантах A/B — при необходимости создаёт каталог и права для Runner. Никакого деплоя приложения и отдельного сервиса не выполняется.

**Конвенция путей (Вариант A):** `/docs/<project-slug>/` на сайте; каталог на диске `/var/www/<main-site>/public/docs/<project-slug>/`; права на запись для gitlab-runner; шаблон CI — «build docs + publish» из общей инфраструктуры.

### Сценарий 2: Сборка, загрузка и деплой (running app)

**Цель:** приложение собирается в CI, загружается на сервер и запускается; доступ по своему домену с SSL.

**Что делается автоматически:**

- Регистрация проекта в оркестраторе: домен/поддомен, тип (frontend, backend, оба), целевой хост.
- DNS: создание A/CNAME для выбранного домена (`manage-dns` или API).
- Traefik: добавление маршрута в `/etc/traefik/dynamic`, reload.
- Директории деплоя: `/var/www/<project-slug>/...`, права для Runner.
- CI: include из общей инфраструктуры; переменные `DEPLOY_PATH`, домены; деплой (build → deploy → перезапуск PM2/сервиса).

**Плюсы:** полный цикл «push → сайт работает», один поток для веб-приложений и API.  
**Минусы:** нужны Runner на целевом хосте, стабильный DNS и Traefik; сложнее, чем «только доки».

### Сценарий 3: Инфраструктура / платформа (deploy later)

**Цель:** проект описывает инфраструктуру (Terraform, Pulumi, Ansible, docker-compose и т.п.) или конфигурацию платформы; сама инфраструктура разворачивается позже (вручную или отдельным пайплайном).

**Что делается автоматически:**

- Репозиторий + при необходимости — сайт документации по сценарию 1.
- CI: шаблон «validate / plan» (lint, terraform validate, dry-run); опционально — артефакты (plan-файлы, упакованные конфиги).
- DNS/Traefik/деплой приложения: не создаются при регистрации; при необходимости отдельный сценарий «запуск окружения» с собственным доменом и деплоем (тогда ближе к сценарию 2).

**Плюсы:** разделение «код инфраструктуры» и «запущенный сервис»; безопаснее для продакшена.  
**Минусы:** «один клик» даёт только подготовленный репо и валидацию, а не работающую платформу — деплой отдельным шагом.

### Сценарий 4: Продукт/утилита — продуктовая страница и артефакты для установки

**Цель:** продукт не как работающий сервис, а как **страница на сайте** с описанием, инструкциями и **артефактами для скачивания** (установщики, пакеты, бинарники). Всё создаётся и обновляется автоматически при релизе.

**Что делается автоматически:**

- **Продуктовая страница:** создаётся/обновляется на основном сайте (например, `/products/<slug>` или поддомен `product.<base>`) с метаданными: название, описание, ссылки на установку, версия.
- **Артефакты:** в CI при теге/релизе — сборка (бинарники, .msi/.deb/.rpm, пакеты), загрузка в хранилище (артефакты GitLab, S3-совместимое, или директория под раздачу); ссылки из артефактов попадают на продуктовую страницу.
- **Документация:** опционально — сценарий 1 (docs) или блок «документация» на той же продуктовой странице.
- DNS: только если продуктовая страница на отдельном поддомене; иначе — маршрут в рамках основного сайта.
- Traefik: не нужен отдельный бэкенд для «страницы продукта», если она часть основного сайта; нужен только раздающий артефакты endpoint (статика или существующий бэкенд).

**Плюсы:** один поток «релиз в репо → продуктовая страница + ссылки на установку»; не нужен деплой приложения как сервиса.  
**Минусы:** нужна интеграция основного сайта с каталогом продуктов (API или генерация страниц из метаданных/артефактов); единое хранилище артефактов и правила именования.

| -------------------------------- | ----------- | ------------------ | --------------------- | ---------------- | ------------------- | ----------------------------------- |
| 1. Только исходники и доки       | да          | да (Pages/статика) | по необходимости      | по необходимости | нет                 | нет                                 |
| 2. Сборка и деплой               | да          | опционально        | да                    | да               | да                  | нет                                 |

| 4. Продукт/утилита               | да          | опционально        | по необходимости      | по необходимости | нет                 | да (страница + ссылки на артефакты) |

Оркестратор и общие шаблоны CI должны поддерживать **выбор типа проекта при регистрации** (или по метке/группе в GitLab), чтобы применять нужный сценарий и не создавать лишние DNS/Traefik/деплой там, где нужны только доки или продуктовая страница.

**Выбор опций через списки (select):** в UI и при проектировании API все ключевые параметры задаются **выбором из списка**, а не вводом с клавиатуры, где это возможно:

- **Тип проекта / сценарий** — выпадающий список: «Только исходники и документация», «Сборка и деплой приложения», «Инфраструктура (deploy later)», «Продуктовая страница + артефакты».
- **Репозиторий GitLab** — список проектов (загрузка через GitLab API), с возможностью поиска/фильтра; опционально — поле ввода URL для внешнего репо.
- **Тип приложения** (при сценарии деплоя) — список: «Frontend», «Backend», «Frontend + Backend».
- **Целевой хост / Runner** (при мульти-сервере) — список зарегистрированных проектов-серверов или тегов Runner.
- **Базовый домен / поддомен** — при необходимости список допустимых базовых доменов и автоподстановка варианта поддомена по имени проекта.

Так пользователь следует по шагам и выбирает из готовых вариантов, что снижает ошибки и ускоряет регистрацию.

## 3B. Как делают конкуренты

Краткий разбор подходов платформ к разным типам публикации — чтобы опереться на проверенные практики и учесть ограничения.

- **GitLab**
  - **Pages:** один тип результата — статический сайт (в т.ч. доки). Отдельного «только репо без сайта» или «продуктовая страница + артефакты» нет; репо и релизы — отдельно.
  - **Releases:** артефакты прикрепляются к тегам вручную или через CI (скрипты/actions). Нет встроенной «продуктовой страницы» на основном домене — только страница репозитория и вкладка Releases.
  - **Deploy:** деплой приложения на свой хост настраивается в CI вручную или через Auto DevOps (Kubernetes); единого «один клик + DNS + Traefik» нет.
  - **Вывод:** нам имеет смысл комбинировать: тип «доки» как аналог Pages; тип «деплой» — свой оркестратор + Traefik/DNS; тип «продукт» — Releases-подобный поток артефактов + собственная продуктовая страница на сайте.
- **GitHub**
  - **Releases + Actions:** типичный паттерн — при создании тега workflow собирает бинарники и загружает их в Release; страница продукта — это страница репозитория и вкладка Releases.
  - **Pages:** только статика (доки/сайт проекта). Нет встроенного «каталога продуктов» с артефактами на основном домене.
  - **Вывод:** сценарий 4 (продуктовая страница + артефакты) у нас может быть сильнее, если страница живёт на основном сайте и автоматически подтягивает ссылки из GitLab Releases/артефактов.
- **Coolify / CapRover (self‑hosted PaaS)**
  - **Coolify:** различает типы приложений (Static, Docker, и т.д.); деплой из репо по выбранному build pack. Нет отдельного сценария «только доки» или «только продуктовая страница с загрузками».
  - **CapRover:** one-click apps — готовые образы; свои приложения — деплой из репо. Нет встроенной модели «продукт = страница + артефакты».
  - **Вывод:** наша дифференциация по сценариям (1–4) и привязка к одному репо по инструкции дают более гибкую модель, чем «всё как деплой приложения».
- **Vercel / Netlify**
  - Шаблоны и деплой из репо; автоматический preview и production. Нет своей DNS/Traefik; нет отдельного сценария «каталог продуктов с артефактами».
  - **Вывод:** идея «один проект — одна инструкция — нужный результат в зависимости от типа» нам подходит; оркестратор и выбор сценария закрывают то, чего у них нет (единый каталог продуктов на своём домене + артефакты).

**Зафиксировано в ТЗ:** поддержка сценариев (1–4) в зависимости от типа проекта; оркестратор и шаблоны CI получают параметр «тип проекта» и выполняют только нужные шаги.

## 3C. Сравнение принятых решений и вариантов (анализ по данным из открытых источников)

Ниже — ключевые решения, варианты по каждому, краткий разбор и выводы на основе практик и статей в интернете. Эти рекомендации перенесены в итоговое ТЗ как обоснование выбранных вариантов.

### 1. Триггер «одного клика»: UI (Вариант A) vs Webhook (Вариант B) vs гибрид

| Критерий     | Вариант A: кнопка в UI                                                                | Вариант B: webhook от GitLab                                                            | Гибрид (A + B)                                                                 |
| ------------ | ------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------ |
| **Контроль** | Явный: пользователь сам нажимает «Зарегистрировать»                                   | Автоматический: push/создание проекта вызывает оркестратор                              | Регистрация по кнопке; обновления (например, повторный онбординг) — по webhook |
| **Ошибки**   | Меньше случайных срабатываний                                                         | Нужны правила (группа, тег), иначе все репо могут триггерить онбординг                  | Гибкость, но сложнее описать поведение                                         |
| **Практики** | UI-триггеры рекомендуют для on-demand, контролируемых деплоев (Pulumi, Harness и др.) | Webhook — стандарт для CI/CD и event-driven автоматизации (Netlify, Google Cloud Build) | Крупные платформы часто дают оба: ручной запуск + триггер по событию           |

**Вывод (зафиксировано в ТЗ):** для MVP — **Вариант A** (кнопка в UI). На следующем шаге — гибрид: webhook на «Pipeline success» или «Project create» для выбранной группы.

### 2. Публикация документации на сайте: путь на основном сайте (A) vs поддомен (B) vs GitLab Pages (C)

**Вывод (зафиксировано в ТЗ):** для сценария «доки появляются на сайте» выбран **Вариант A** (каталог на основном сайте `/docs/<slug>/`): лучше для SEO, один домен, проще эксплуатация. Вариант B — запасной при требовании отдельного поддомена. Вариант C допустим при готовности к внешним ссылкам на Pages.

### 3. Оркестратор: отдельный сервис vs расширение Management UI

**Вывод (зафиксировано в ТЗ):** при текущем объёме — **расширение Management UI** (новые endpoints + фоновые задачи). Отдельный сервис — при росте числа интеграций или нескольких типов триггеров.

### 4. Мульти-сервер: один Traefik vs Traefik на каждой VM

**Вывод (зафиксировано в ТЗ):** для мульти-серверной фазы — **модель «один Traefik, несколько бэкендов»**: один entry point, сервисы на `http://<vm-internal-ip>:port`. Traefik на каждой VM — только при явной изоляции по зонам.

### 5. Один проект инфраструктуры для всех серверов (вместо «проект на сервер»)

**Вывод (зафиксировано в ТЗ):** принята модель **один репозиторий для конфигурации всех серверов**. Плюсы: исправления ошибок и обновление скриптов/шаблонов в одном месте, не нужно переносить изменения между несколькими проектами; один MR/коммит затрагивает всю инфраструктуру. Структура репо — по п. 2.3 (общее в корне/`scripts`/`templates`, конфиг по хостам в `config/servers/<id>/`). Секреты — только через переменные CI или внешнее хранилище.

### 6. Выбор опций через списки (select) в форме регистрации

**Вывод (зафиксировано в ТЗ):** выпадающие списки для типа проекта, типа приложения, целевого хоста, базового домена. Для репозитория — поиск/автодополнение по списку из GitLab API + опция «другой URL» с полем ввода.

## 4. Компоненты и изменения

### 4.1 GitLab и общая инфраструктура

- **Два типа репозиториев:** (1) проекты-приложения — отдельные репо с кодом сайта/API; (2) единый проект инфраструктуры — один репо со скриптами, шаблонами CI и конфигом всех серверов (п. 2.2–2.3). Приложения подключают include из единого проекта инфраструктуры.
- **Шаблон пайплайна для приложений:** общий `.gitlab-ci.yml` (include из единого проекта инфраструктуры, каталог `templates/`), чтобы все проекты-приложения могли им пользоваться. Учитывать: разный тип приложений (только frontend, frontend+backend, статика). При необходимости в том же репо — пайплайн для применения конфигов по серверам (по путям `config/servers/<id>/**`).
- **Переменные CI:** для каждого проекта-приложения — как минимум `DEPLOY_PATH`, опционально `FRONTEND_DOMAIN` / `BACKEND_DOMAIN`, при мульти-сервере — тег Runner (соответствует одному из серверов из `config/servers/`). Задавать вручную или через GitLab API из оркестратора при регистрации.
- **Webhook (при гибридном варианте):** в настройках проекта/группы — Pipeline events или Project create → URL оркестратора; оркестратор по payload определяет проект и запускает онбординг.

Ссылки: GitLab [Custom project templates](https://docs.gitlab.com/ee/user/group/custom_project_templates.html), [include](https://docs.gitlab.com/ee/ci/yaml/includes.html), [Webhooks](https://docs.gitlab.com/user/project/integrations/webhook_events/).

### 4.2 «Deploy Orchestrator» (новый компонент)

- Отвечает за: создание DNS (вызов `manage-dns` или локальный DNS API), генерация и запись YAML в `/etc/traefik/dynamic`, создание директорий деплоя, при необходимости — вызов GitLab API (файл в репозитории, CI variables). Список доступных хостов оркестратор берёт из единого проекта инфраструктуры (например из `config/servers/` или из своей базы, синхронизированной с ним).
- Вход: идентификатор проекта (path/ID), желаемый домен/поддомен, **тип проекта/сценарий** (1–4 из п. 3A: только доки, деплой приложения, инфраструктура, продуктовая страница + артефакты), тип приложения при деплое (frontend, backend, оба), целевой хост — тег Runner или идентификатор сервера из единого проекта (для мульти-сервера).
- **Реализация (зафиксировано в ТЗ):** расширение Management UI — новые API endpoints и фоновые задачи, вызывающие существующие скрипты и `manage-dns`. Отдельный сервис — при росте числа интеграций.

### 4.3 Traefik

- Оставляем file provider и watch. Оркестратор только добавляет/обновляет/удаляет файлы в `/etc/traefik/dynamic` (по одному файлу на проект или frontend/backend), затем `systemctl reload traefik`. Без изменения глобального [traefik.yml](../config/single-machine/traefik/traefik.yml).
- Правила: как сейчас — `Host(\`subdomain.base.domain\`)`, сервис на `http://<internal-ip>:<port>`. Для мульти-сервера: internal-ip будет разный в зависимости от того, на каком хосте крутится приложение.

### 4.4 DNS

- Без смены провайдера: продолжать использовать `manage-dns create/delete` и при необходимости локальный DNS API. Оркестратор вызывает их при регистрации/удалении проекта. Один базовый домен для всех проектов или несколько (уже есть [manage-base-domains.sh](../scripts/single-machine/manage-base-domains.sh)).

### 4.5 CI/CD на стороне Runner и общая инфраструктура

- Структура деплоя: вместо единственного пути `/var/www/borisovai-site` ввести правило: например `/var/www/<project-slug>/frontend` и `/var/www/<project-slug>/backend` (или один каталог на проект). [setup-cicd.sh](../scripts/single-machine/setup-cicd.sh) и общий шаблон `.gitlab-ci.yml` опираются на переменную `DEPLOY_PATH` (уже есть), которая для каждого проекта задаётся разная.
- Runner: на каждом сервере — свой Runner с уникальным тегом (например `deploy-vm1`, `deploy-vm2`). Соответствие тег ↔ сервер хранится в едином проекте инфраструктуры (`config/servers/<id>/env.yml`). Проекты-приложения в CI указывают тег Runner целевого хоста (или переменную из оркестратора).
- Все проекты-приложения подключают шаблоны CI из единого проекта инфраструктуры (include); изменения в скриптах и шаблонах вносятся в одном репо.

### 4.6 Management UI

- Уже есть создание сервиса (DNS + Traefik). Добавить сценарий «Зарегистрировать проект GitLab» с **выбором из списков (select):** тип проекта (сценарий 1–4), репозиторий (список из GitLab API), при необходимости домен, тип приложения при деплое, при мульти-сервере — целевой хост (тег Runner) из списка, источник списка — единый проект инфраструктуры (`config/servers/` или синхронизированная с ним база); вызов оркестратора с выбранными значениями. Все ключевые параметры — выбор из списка (п. 3A). При желании — список проектов, уже зарегистрированных для деплоя (маппинг проект → домен, путь, хост). Опционально: сценарий «Добавить новый сервер» — вывод чеклиста (создать каталог в едином репо, заполнить `env.yml`, установить Runner на VM, зарегистрировать тег); без создания нового GitLab-проекта.

### 4.7 Мульти-сервер (новые VM) и единый проект инфраструктуры

- **Один проект инфраструктуры для всех серверов (п. 2.2–2.3).** Конфиг каждого хоста хранится в подкаталоге `config/servers/<id>/`; новый сервер добавляется созданием такого каталога и файла `env.yml` (internal_ip, runner_tag, base_domain), без создания нового GitLab-проекта. На новой VM устанавливается Runner с уникальным тегом; оркестратор при необходимости получает список хостов из этого репо (или из синхронизированной базы).
- **Регистрация новой VM:** (1) в едином репо — каталог `config/servers/<vm-id>/`, `env.yml`; (2) на VM — установка пакетов, Runner с тегом, ключи; (3) при необходимости — регистрация в оркестраторе. **Модель Traefik (зафиксировано в ТЗ):** один инстанс Traefik, все маршруты в нём, бэкенды на разных VM (`http://<vm-internal-ip>:port`).
- **Модель «один Traefik, несколько бэкендов»:** все маршруты в одном Traefik, сервисы указывают на `http://<vm-internal-ip>:port`. Требуется стабильная внутренняя сеть и знание IP каждой VM; IP и тег хранятся в `config/servers/<id>/env.yml` единого проекта.
- **Альтернатива «Traefik на каждой VM»:** на входе один балансировщир (или DNS с разными A-записями), на каждой VM — свой Traefik; оркестратор обновляет конфиг только на нужной VM (по SSH или локальный API). Использовать только при явной изоляции по зонам.
- **Чеклист подключения новой VM:** создать `config/servers/<id>/` и `env.yml` в едином репо, установить пакеты и Runner на VM, при необходимости зарегистрировать в оркестраторе.

## 5. Риски и сложности

- **Права и безопасность:** оркестратор должен вызывать `manage-dns` и писать в `/etc/traefik/dynamic` — нужны права (sudo/отдельный пользователь с ограниченными возможностями). Не хранить токены GitLab в открытом виде.
- **Идемпотентность:** повторная регистрация одного и того же проекта не должна ломать DNS и Traefik; обновление домена или порта — предсказуемо.
- **Откат:** удаление проекта должно снимать маршрут Traefik, при необходимости DNS и не оставлять «мусор» в `/var/www`. При отключении VM — снятие сервисов с этого хоста и при желании удаление или архивирование каталога `config/servers/<id>/` в едином репо.
- **Ограничения GitLab:** создание файла в репозитории и CI variables через API требует токена с правами; шаблоны групп — в Premium/Ultimate. Для CE — опереться на include из единого проекта инфраструктуры (каталог `templates/`) или один раз вручную добавить в проект приложения минимальный `.gitlab-ci.yml` с include.
- **Мульти-сервер и единый проект:** согласованность между `config/servers/<id>/env.yml` и реальной VM (тег Runner, IP). Мульти-сервер — фаза 2 после стабильного «одного клика» на одном сервере; структура единого проекта (п. 2.3) закладывается в фазе 1.

## 6. Этапы реализации

- **Фаза 1 — MVP (один сервер) и задел под единый проект инфраструктуры:**
  - Ввести **единый проект инфраструктуры** (п. 2.3): в текущем репо или выделенном — каталоги `templates/` (шаблоны CI для приложений), при желании `config/servers/vm1/` с `env.yml` для первого хоста, чтобы структура была готова к мульти-серверу.
  - Внедрить оркестратор как **расширение Management UI**: регистрация проекта-приложения по имени/домену, создание DNS, Traefik, директорий, подстановка шаблона CI (include из единого проекта) и переменных через API.
  - Общий шаблон `.gitlab-ci.yml` для приложений (include из `templates/`), один Runner.
  - Документация: как завести новый проект «в один клик» через UI (или один вызов API). Структура единого проекта заложена так, чтобы в фазе 2 добавлять серверы подкаталогами без нового репо.
- **Фаза 2 — Мульти-сервер при одном проекте инфраструктуры:**
  - В едином репо: для каждой новой VM — каталог `config/servers/<id>/` и `env.yml` (internal_ip, runner_tag, base_domain).
  - Подключение второй VM: создать каталог в едином репо, на VM установить Runner с уникальным тегом; в оркестраторе при регистрации приложения — выбор целевого хоста (тег Runner) из списка, источник списка — `config/servers/` или синхронизированная база.
  - Traefik: один инстанс с бэкендами на всех VM (зафиксировано в п. 4.7).
  - Чеклист и скрипт «добавить новую VM» (создание каталога, `env.yml`, установка Runner) в репозиторий.
- **Фаза 3 — Оптимизация и агенты (по желанию):**
  - Использование агентов (например, в Cursor/n8n) для автоматического заполнения настроек проекта, генерации минимального `.gitlab-ci.yml` или подсказок по переменным.
  - Опционально: автоматическая регистрация по webhook при создании репозитория в определённой группе.

## 7. Критерии приёмки Фазы 1

- Единый проект инфраструктуры (п. 2.3) создан: каталог `templates/` с шаблонами CI доступен для include приложениям.
- В Management UI реализован сценарий «Зарегистрировать проект GitLab» с выбором типа проекта (1–4), репозитория (список из GitLab API), домена, типа приложения при деплое; вызов оркестратора выполняет: DNS (при необходимости), Traefik (при сценариях 2/4 с деплоем), создание директорий, подстановку CI (include из единого проекта) и переменных через GitLab API (при наличии токена).
