## 🎯 Что можно автоматизировать?

### ✅ Уровень 0: Полностью автоматизировать (без одобрения)

```yaml
Мониторинг и диагностика:
  - ✅ Проверка здоровья VM (CPU, память, диск)
  - ✅ Проверка доступности сети
  - ✅ Проверка дисков на ошибки
  - ✅ Анализ логов системы
  - ✅ Мониторинг трафика

Очистка:
  - ✅ Удалять файлы > 30 дней из /tmp
  - ✅ Ротация логов (старые сжимать, очень старые удалять)
  - ✅ Очищать кэши пакетов (apt clean)
  - ✅ Удалять неиспользованные Docker образы
  - ✅ Удалять неиспользованные виртуальные диски

Резервные копии:
  - ✅ Создавать ежедневные снимки VM (snapshots)
  - ✅ Ротация бэкапов (хранить последние 30 дней)
  - ✅ Проверять целостность бэкапов
  - ✅ Удалять старые бэкапы по расписанию
```

### ⚠️ Уровень 1: Автоматизировать, но с проверкой (Auto-approve)

```yaml
Обновления ОС:
  - ⚠️ Установить security updates (безопасность)
  - ⚠️ Установить critical bug fixes
  - ⚠️ НЕ обновлять kernel (требует перезагрузку)

Обновления пакетов:
  - ⚠️ Обновить Docker (если не критично)
  - ⚠️ Обновить инструменты (Terraform, Ansible, etc.)
  - ⚠️ НЕ обновлять БД (требует миграция)

Управление дисками:
  - ⚠️ Расширить диск если > 80% заполнен
  - ⚠️ Добавить новый диск если нет места
  - ⚠️ Сжать диск если < 50% использования

Масштабирование:
  - ⚠️ Добавить CPU если загрузка > 80% в течение часа
  - ⚠️ Добавить памяти если свободно < 10%
  - ⚠️ Добавить хранилища если < 15% свободно
```

### 🔒 Уровень 2: Требует одобрения человека

```yaml
Критичные обновления:
  - 🔒 Обновление ядра (требует перезагрузка)
  - 🔒 Обновление БД (может нарушить совместимость)
  - 🔒 Major версии приложений

Миграции:
  - 🔒 Миграция VM на другой хост Proxmox
  - 🔒 Миграция данных в другое облако

Мониторинг и конфигурация:
  - 🔒 Изменение security rules (firewall)
  - 🔒 Изменение network configuration
  - 🔒 Изменение backup policy
```

## 🏗️ Архитектура: Agent + Proxmox/Облако

### Proxmox Integration

```
┌─ Agent (Claude 4)
│  ├─ Reads via Proxmox API (read-only)
│  │  ├─ GET /api/nodes/{node}/status
│  │  ├─ GET /api/nodes/{node}/storage
│  │  ├─ GET /api/nodes/{node}/qemu
│  │  └─ GET /api/nodes/{node}/lxc
│  │
│  ├─ Executes via Proxmox API
│  │  ├─ POST /api/nodes/{node}/qemu/{vmid}/status/reboot
│  │  ├─ POST /api/nodes/{node}/storage/{storage}/content
│  │  ├─ POST /api/nodes/{node}/qemu/{vmid}/resize
│  │  └─ POST /api/nodes/{node}/firewall/rules
│  │
│  └─ Can execute shell commands via Proxmox
│     ├─ apt update && apt upgrade (в VM)
│     ├─ df -h (проверка диска)
│     ├─ docker prune (очистка)
│     └─ systemctl restart service

└─ Proxmox
   ├─ VMs (Debian 12 с приложениями)
   ├─ LXC контейнеры (если нужны)
   └─ Storage (местные диски или NAS)
```

### Облако Integration (DigitalOcean)

```
┌─ Agent (Claude 4)
│  ├─ Reads via API (read-only)
│  │  ├─ GET /v2/droplets (список VMs)
│  │  ├─ GET /v2/droplets/{id}/snapshots (снимки)
│  │  └─ GET /v2/account (статус аккаунта)
│  │
│  ├─ Executes via API
│  │  ├─ POST /v2/droplets/{id}/actions (power, reboot, resize)
│  │  ├─ POST /v2/droplets/{id}/snapshots (создать снимок)
│  │  ├─ DELETE /v2/snapshots/{id} (удалить старый снимок)
│  │  └─ POST /v2/firewalls (update rules)
│  │
│  └─ SSH в VM для команд
│     └─ apt update, docker prune, systemctl restart, etc.

└─ DigitalOcean Account
   ├─ Droplets (VMs)
   ├─ Snapshots (бэкапы)
   ├─ Firewalls (сетевые правила)
   └─ Billing (отслеживание расходов)
```

## 📋 Конкретные задачи обслуживания

### Задача 1: Автоматическое обновление ОС

```yaml
Trigger: Каждый день в 2:00 AM (низкая нагрузка)

Процесс:
  1. Agent проверяет: есть ли доступные обновления?
     └─ apt list --upgradable (в VM)

  2. Если есть только security updates:
     ├─ Auto-approve
     ├─ apt update && apt upgrade -y
     ├─ Проверить что всё работает
     └─ Залогировать

  3. Если есть kernel update:
     ├─ Требует approval
     ├─ Отправить уведомление
     ├─ Подождать 24 часа
     ├─ Если одобрено: обновить + перезагрузить
     └─ Если не одобрено: пропустить

  4. After completion:
     ├─ Health check
     ├─ Если broken: automatic rollback (снимок)
     └─ Уведомить результат

Cost: ~$0.001 (minimal API calls)
```

### Задача 2: Управление дисковым пространством

```yaml
Trigger: Каждый час (или когда заполнение > 80%)

Процесс:
  1. Проверить использование диска
     └─ df -h (в VM)

  2. Если < 90%:
     └─ Всё хорошо, продолжить

  3. Если 90-95%:
     ├─ Автоматически очистить:
     │  ├─ apt clean (удалить старые пакеты)
     │  ├─ docker system prune (удалить неиспользованные)
     │  ├─ rm /var/log/old_logs.gz (очень старые логи)
     │  └─ Переproверить

  4. Если > 95%:
     ├─ Требует approval
     ├─ Опции:
     │  ├─ Расширить диск (в Proxmox/облаке)
     │  ├─ Перейти на большую VM
     │  └─ Очистить пользовательские данные
     └─ Ожидать решения

Cost: ~$0 (локальные команды)
```

### Задача 3: Автоматические снимки (Snapshots)

```yaml
Trigger: Ежедневно в 3:00 AM

Proxmox вариант:
  1. Создать снимок VM
     └─ proxmox API: POST /api/nodes/{node}/qemu/{vmid}/snapshot

  2. Ротация снимков (удалить старше 30 дней)
     └─ API: DELETE /api/nodes/{node}/qemu/{vmid}/snapshot/{snap}

  3. Проверить целостность
     └─ Попробовать восстановить в тестовом VM
     └─ Если ОК: удалить тестовый VM
     └─ Если fail: Alert

DigitalOcean вариант:
  1. Создать снимок Droplet
     └─ API: POST /v2/droplets/{id}/snapshots

  2. Ротация (удалить старше 30 дней)
     └─ API: DELETE /v2/snapshots/{id}

  3. Стоимость: Снимки стоят $0.05/GB в месяц
     └─ Для 50GB: $2.50/месяц

Cost: Дешевле чем потерять данные!
```

### Задача 4: Масштабирование при нагрузке

```yaml
Trigger: Каждые 5 минут (мониторинг)

Proxmox:
  1. Проверить CPU/Memory Proxmox хоста
     └─ Если CPU > 80% или Memory > 85%:

  2. Опции масштабирования:
     ├─ Добавить CPU VM (горячее добавление)
     ├─ Добавить память VM (требует перезагрузка)
     ├─ Перенести VM на другой хост (миграция)
     └─ Добавить новый хост Proxmox

  3. Для Tier 2: Auto-approve если:
     ├─ CPU + 2 ядра (осталось место на хосте)
     ├─ Memory +2GB (осталось на хосте)
     └─ Требует одобрение если нужен новый хост

DigitalOcean:
  1. Проверить CPU/Memory Droplet
     └─ Если > 80%: требует action

  2. Опции:
     ├─ Vertical scaling (новый размер Droplet)
     │  └─ Требует ~1 мин downtime
     ├─ Horizontal scaling (добавить новый Droplet)
     │  └─ Требует load balancer
     └─ Требует одобрение (cost impact)

Cost: Может быть дорого (новый Droplet = +$50/месяц)
```

### Задача 5: Проактивный мониторинг здоровья

```yaml
Trigger: Каждые 5 минут (continuous)

Проверки:
  1. Хост доступен (ping)
     └─ Если timeout: Alert, попробовать SSH

  2. ОС работает нормально
     └─ uptime, load average, processes

  3. Диски работают нормально
     └─ SMART статус (в Proxmox: smartctl)
     └─ Если warning: Alert + план замены

  4. Сеть работает нормально
     └─ Ping шлюза, проверка DNS, проверка интернета

  5. Сервисы работают
     └─ systemctl status nginx/postgres/redis
     └─ Если down: restart + alert

  6. Бэкапы работают
     └─ Проверить последний снимок (не старше 25 часов)
     └─ Если старше: Create new snapshot

Cost: ~$0 (локальные проверки)
```

## 🔐 Безопасность автоматизации инфраструктуры

### Риски

```
Риск 1: Agent может сломать всю инфру
  → Решение: Dry-run все критичные операции

Риск 2: Agent может масштабировать и потратить много денег
  → Решение: Daily/Monthly budget limit в облаке

Риск 3: Agent может удалить важные снимки
  → Решение: Минимум 3 копии, auto-archive на S3

Риск 4: Agent может изменить firewall rules
  → Решение: Только одобренные правила из конфига
```

### Guardrails для инфраструктуры

```yaml
Rate limiting:
  - Max 10 операций в час (не спамить API)
  - Max 5 одновременных операций

Resource limits:
  - Max resize CPU: +4 ядра за раз
  - Max resize RAM: +8GB за раз
  - Max новых Droplets: 1 в день (cost control)

Cost controls:
  - Daily limit: $10 (alert при $8)
  - Monthly limit: $200 (stop при $180)
  - Каждая операция:估算 cost перед выполнением

Approval thresholds:
  - Security updates: Auto
  - Bug fixes: Auto
  - Kernel update: Требует одобрение
  - Масштабирование > 50%: Требует одобрение
  - Новый хост: Требует одобрение
```

## 🚀 Реализация по фазам

```
✅ Базовый мониторинг (health check)
✅ Логирование событий
✅ Очистка временных файлов
✅ Создание снимков VM (manual trigger)

Complexity: Low
Effort: 40 часов
```

```
✅ Автоматизация обновлений ОС (security only)
✅ Автоматическое управление диском

✅ Health check с auto-restart
✅ Alerts + notifications

Complexity: Medium
Effort: 60 часов

### Phase 3: Advanced (неделя 5-8)
```

✅ Cost optimization

Complexity: High
Effort: 80 часов
